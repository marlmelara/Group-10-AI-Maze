\section*{Introduction}

This report describes the second check-in for the maze solver project. At this stage, the main focus was building a working maze environment from the provided image, solving the maze without hazards, visualizing the solution path, and then integrating hazards into the environment.

The system now supports a full grid-based maze environment with movement constraints, pathfinding, hazard interaction, and visualization of both the maze layout and the agent's behavior.

\section*{Maze Representation}

The maze is represented internally as a 64$\times$64 logical grid. Each logical cell corresponds to a valid position the agent can occupy, while walls are stored as blocked movement edges between neighboring cells instead of as filled wall cells.

Two arrays are used to store the wall structure:

\begin{itemize}[itemsep=0.2em]
    \item \texttt{maze\_walls\_right.npy} --- stores whether movement from a cell to the cell on its right is blocked
    \item \texttt{maze\_walls\_down.npy} --- stores whether movement from a cell to the cell below it is blocked
\end{itemize}

This representation preserves the original maze layout while allowing efficient movement checks and graph-based search.

\section*{Maze Solving (BFS)}

A Breadth-First Search (BFS) algorithm was used to solve the maze from the start position to the goal position.

BFS was selected because the maze is an unweighted grid environment, so BFS guarantees a shortest path in terms of number of moves. It is also simple, deterministic, and well suited for validating whether the maze representation is correct.

The solver produced the following results:

\begin{itemize}[itemsep=0.2em]
    \item Path length: 423 cells
    \item Total moves required: 422
    \item Turns required (maximum of five actions per turn): 85
\end{itemize}

These results confirmed that the reconstructed maze was connected correctly and that the movement logic in the environment was functioning as expected.

\section*{Hazard System}

After validating the maze and solver, hazards were added to the environment to introduce more complex behavior. Three hazard types were implemented:

\begin{itemize}[itemsep=0.3em]
    \item \textbf{Death pits} --- the agent immediately dies upon entering the cell and respawns at the start location on the next turn
    \item \textbf{Teleport pads} --- the agent is moved instantly to a predetermined destination cell
    \item \textbf{Confusion pads} --- the agent's controls are inverted for the remainder of the current turn and the following turn
\end{itemize}

Hazards were generated automatically at approximately 7\% of navigable cells, which satisfies the required density range of 5--10\%. During hazard generation, a valid path between the start and goal positions was preserved so that the maze remained solvable.

\section*{Environment Behavior}

The hazard system was validated through scripted demonstrations showing each hazard type in action. These demonstrations included:

\begin{itemize}[itemsep=0.2em]
    \item stepping onto a death pit and respawning at the start
    \item stepping onto a teleport pad and relocating to its mapped destination
    \item stepping onto a confusion pad and observing temporarily inverted movement
\end{itemize}

The environment follows the required project turn format:

\begin{itemize}[itemsep=0.2em]
    \item each turn accepts between 1 and 5 actions
    \item actions are executed sequentially
    \item remaining actions are ignored if the agent dies or reaches the goal
    \item the environment returns a \texttt{TurnResult} object containing the resulting state information
\end{itemize}

This behavior allows the environment to function as a complete grid-world simulation rather than only as a static maze solver.
